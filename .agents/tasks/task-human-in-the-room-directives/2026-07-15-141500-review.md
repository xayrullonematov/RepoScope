# Human-in-the-Room Directives: event type, state projection, prompt injection, API, and Team Room UI

This change adds a 14th event type (`human-directive`) to the event-sourced architecture, allowing humans to issue persistent free-text directives that flow into all subsequent agent LLM calls. The implementation spans the full vertical: domain types, state projector handler, context assembler integration at priority 3, prompt builder formatting, a POST API route with Zod validation, four new UI components (TeamRoomPanel, DirectivePanel, DirectivesList, TeamActivityFeed), and property-based tests for the projector. The approach cleanly separates directives from existing `user-intervention` constraints â€” directives are session-wide standing instructions rather than round-scoped context.

**Watch for:** The state projector always appends directives (including deactivation events with `active: false`) rather than filtering them out, diverging from the design doc and making `DirectivesList`'s `.filter(d => d.active)` the sole defense against showing stale directives (confirmed). The consensus prompt omits human directives entirely â€” agents in the synthesis stage won't see them (confirmed). The API route has no session-existence check, so posting a directive for a nonexistent session silently writes an orphan event (likely).

## High-level view

The state projector handler unconditionally appends every `human-directive` event to the `humanDirectives` array, including events where `active: false`. The design doc specifies that deactivation should remove the directive from projected state, but the implementation relies on downstream filters (UI, prompt builder) to ignore inactive entries. Any new consumer that forgets to filter will surface stale directives.

The prompt builder injects directives into the stable system block for proposal, critique, and revision prompts via `buildStableSystemBlock`. However, `buildConsensusPrompt` is a standalone function that builds its own system prompt without calling `buildStableSystemBlock`, so the consensus synthesizer never sees human directives. A directive like "do not recommend Kubernetes" won't be visible during synthesis.

The API route validates input well but does not verify that the target `sessionId` corresponds to an existing session before appending. The existing `intervene/route.ts` pattern checks session existence; this route skips that step. The DELETE endpoint described in the design doc is unimplemented, leaving directives permanently active.

The TeamActivityFeed displays the *current* directive count on historical events, so a directive added after a proposal retroactively appears to have influenced it.

## State projector: append-only semantics vs design-doc contract

The design doc specifies two branches in `handleHumanDirective`: if `active` is false, the directive is removed from the projected array; otherwise it's added or replaced. The implementation does neither removal nor replacement:

```typescript
function handleHumanDirective(state, content, event): SessionState {
  const directive: HumanDirective = { ... active: (content.active as boolean) ?? true };
  return { ...state, humanDirectives: [...state.humanDirectives, directive] };
}
```

Replaying "add A, add B, deactivate A" produces `[A(active:true), B(active:true), A(active:false)]` in `state.humanDirectives`. The prompt builder defensively filters with `.filter(d => d.active)`, and `DirectivesList` does the same, so user-visible behavior is currently correct. But the design doc's contract (projected state = only active directives) is the safer invariant â€” it pushes correctness into the single projector function rather than distributing it to N consumers.

There's no test covering the deactivation path (appending an `active: false` event and asserting it's removed from projected state), which is consistent with the implementation but leaves the gap untested.

## Consensus prompt blind spot

The prompt builder has two construction paths. `buildProposalPrompt`, `buildCritiquePrompt`, and `buildRevisionPrompt` all call `buildStableSystemBlock(agent, context)`, which includes the "Human Team Directives" section. `buildConsensusPrompt` constructs its system prompt inline:

```typescript
const systemPrompt = [
  `You are the Consensus Synthesizer.`,
  ...
  `## Active Constraints`,
  formatConstraints(context.constraints),
  ...
].join("\n");
```

No `formatHumanDirectives` call appears. The consensus synthesizer merges four agents' positions into agreements and decisions. If a directive constrains the final architecture, the synthesizer may produce a consensus that contradicts it because it never saw the instruction. This undermines the feature's session-wide guarantee.

## API route: missing session guard and missing DELETE

The route file exports only `POST`. A POST to `/api/sessions/nonexistent-id/directives` will call `eventStore.appendEvent` with a `sessionId` that has no corresponding `Session` record. If the Prisma schema enforces a foreign-key relation, it'll throw a 500 with a generic "Failed to add directive" message rather than a clear 404.

The design doc specifies a DELETE endpoint for deactivation and `DirectivesList` is designed to show a deactivate button. Neither is implemented. The `active` field on the interface is write-once â€” directives accumulate permanently for the session's lifetime.

## Token budget: all-or-nothing truncation

The context assembler truncates directives and constraints together:

```typescript
if (totalTokens() > budget) {
  finalConstraints = [];
  finalHumanDirectives = [];
}
```

A session that accumulates many directives (especially without deactivation) will hit the boundary and lose all directives simultaneously rather than shedding the oldest first. Combined with the missing rate limit on directive creation, a determined user can push the context over budget and cause complete directive loss.

## TeamActivityFeed: retroactive directive counts

The activity feed computes `activeDirectiveCount` from the current `humanDirectives` prop (final projected state), then applies that count to every historical event: "Senior Engineer completed proposal with 3 active directives." At the time that proposal was generated, there may have been 0 or 1 directive. This produces misleading timeline entries â€” a directive added after a proposal will retroactively appear to have influenced it. The fix is to compute a running count by interleaving directive events with other events chronologically.
