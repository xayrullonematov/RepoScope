# Phase 3: Efficiency-vs-Baseline Metric — Implementation Prompt

## Context

The Agent Society track rubric explicitly requires “measurable efficiency gain over single-agent baseline.” Every other claim in this project (real tool grounding, persona-based task decomposition, disagreement resolution) is now backed by verified evidence — this is the one remaining requirement with zero evidence behind it.

**The honesty constraint for this entire phase: report the real result, even if it’s not a clean win.** A multi-agent system costs significantly more tokens than one call (you already have proof of this — Phase 1’s single proposal round spent ~90k input/6k output tokens). If the comparison doesn’t show a clear quality advantage proportional to that cost, that’s still a useful, honest finding — don’t tune the comparison to manufacture a win. An accurate “here’s the real tradeoff” claim is worth more to judges (and more defensible under scrutiny) than an inflated one.

## Before You Start

Inspect before assuming:

- `round-orchestrator.ts` — confirm how the Proposal stage’s repo-grounding context (`repoCtx`, tool loop) is invoked, so the baseline can reuse the exact same grounding mechanism
- The structured output schema used for risks/decisions in agent responses — the baseline must use the *same schema*, or the comparison is meaningless
- `agent-tool-loop.ts` — confirm whether it can be invoked persona-agnostically (no system-prompt persona injection) for the baseline run

## Objective

Run the same problem + same repo through (a) the existing 4-agent debate pipeline and (b) a single ungrounded-by-persona LLM call with identical tool access, then report a defensible, honest comparison.

## Design Principle: isolate one variable

The baseline must have **the same repo-grounding tool access** (`read_file`, `list_files`, `search_code`) as the multi-agent system. If the baseline doesn’t get tools, you’re measuring “does grounding help” (already proven in Phase 1), not “does multi-agent debate help beyond grounding” (what this phase needs to prove). Only the multi-persona/multi-round structure should differ between the two arms.

## Tasks

### 1. Build the baseline runner

- New file/script: `src/lib/baseline-runner.ts` (or a standalone script under `scripts/` if that fits existing conventions better — check)
- Single LLM call, same model/provider as the agents, with the same three tools and the same call cap (6) as a single agent gets
- Prompt: explicitly ask it to produce a full engineering assessment covering architecture, security, performance, AND product considerations — this is a fair, non-strawman baseline, not a generic “any thoughts?” prompt
- **Output the same structured schema** used by the debate agents for decisions/risks, so results are directly comparable, not eyeballed

### 2. Define the comparison metric

- Primary metric: count of distinct, non-overlapping risk/decision categories surfaced — baseline vs. the union across all 4 debate agents’ final consensus output
- Secondary, must-report-alongside-it: token/cost overhead — total tokens spent across the full 4-agent debate vs. the single baseline call. This is the “cost” side of the efficiency claim; reporting the win without the cost is misleading.
- Optional stretch, only if time allows: a coverage check — does the baseline’s single answer happen to touch all 4 lenses (architecture/security/performance/product), or does it cluster on 1-2? This is often where single-agent answers fall short even when verbose.

### 3. Build the comparison report

- New file: `src/lib/baseline-comparison.ts` (or wherever generates a comparison artifact)
- Output: a clean JSON/markdown report — categories caught by both, categories caught only by the debate system, categories caught only by baseline (report this honestly even if it happens), token cost for each arm
- Run this against the same repo(s) already used for Phase 1 validation (the SaaS starter repo with real auth/webhook/billing surface area) so the result is comparable to evidence you already have

### 4. Decide surfacing scope

- Minimum bar: a script/test harness that produces the comparison report for your submission writeup — this alone satisfies the rubric requirement and is lower-risk given remaining time
- Stretch goal, only if time allows: a small UI element showing the comparison (e.g., “4-agent debate caught N risk categories vs. M for a single pass”) — valuable for the demo, but don’t let this block finishing the metric itself

## Constraints

- Do not cherry-pick the repo or problem to favor the multi-agent result — use the same test cases already validated in Phase 1.
- Do not let the baseline’s tool access differ from what a single debate agent gets (same call cap, same tools).
- Report token/cost overhead in the same artifact as the quality comparison — they’re two halves of one “efficiency” claim, not separate facts.

## Acceptance criteria

- [ ] Baseline runner produces structured output using the same schema as debate agents
- [ ] Baseline has identical tool access and call cap to a single debate agent
- [ ] Comparison report shows category overlap/difference, computed deterministically (not eyeballed)
- [ ] Token/cost numbers for both arms are in the same report
- [ ] Result is reported as-is, including if it doesn’t favor the multi-agent system
- [ ] Comparison run against at least one repo already used in Phase 1 (for continuity of evidence)

## Out of scope for this pass

- UI surfacing of the comparison (stretch goal only, not required)
- Running the comparison across many repos/problems — one solid, honest comparison is sufficient for the submission
- LLM-as-judge subjective quality scoring (adds cost and a new dependency; the category-count + coverage metric is sufficient and more defensible)
